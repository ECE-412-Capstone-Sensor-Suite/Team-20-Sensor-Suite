Description

Command Line Interface (CLI) serial port device.

Note
Use the Device API to control this device. This page details the parameters to pass to the functions of that API.
This documents the "low-level" CLI device. The CLI module wraps this CLI device into higher layer functions, and is usually simpler to use than this device.

The CLI device is responsible for:

  • Maintaining the CLI shell.
  • Accepting CLI commands from the user and dispatching notifications to the appropriate command handlers.
  • Print output, possibly by those handlers.

When a user types a command on the CLI port (possibly with parameters) and presses Enter, this device generates a dn_cli_notif_type_t "notification" to the corresponding application through a channel corresponding to this command. Before your application can receive these notifications, it needs to register each command with this device.

The following table indicates which Device API functions apply to this device. Calling one of the unsupported functions on this device returns DN_ERR_NOT_SUPPORTED. Only the supported functions are detailed below.

dn_open dn_ioctl dn_read dn_write dn_close
YES YES NO YES YES

Open the device

You need to call dn_open() to open this device before using any of the other functions. To open this device, use the dn_open() function with the following parameters:

Parameter Name Parameter Value
device DN_CLI_DEV_ID
args Pointer to a pre-populated dn_cli_open_args_t structure.
argLen Size of dn_cli_open_args_t.

Registering Commands

Your application needs to register each CLI commands it will handle.

To register a command, your application needs to call dn_ioctl() with the following parameters:

Parameter Name Parameter Value
device DN_CLI_DEV_ID
request DN_IOCTL_CLI_REGISTER
args Pointer to a pre-populated dn_cli_registerCmd_t structure.
argLen Size of dn_cli_registerCmd_t.

The dn_cli_registerCmd_t struct specifies (among others):

  • The channel on which the CLI device should send the string input by the user.
  • A unique identifier for each command.

CLI output

To print arbitrary characters over CLI, your application needs to call the dn_write() function with the following parameters:

Parameter Name Parameter Value
device DN_CLI_DEV_ID
buf A pointer to a buffer containing the bytes to print.
bufSize The number of bytes to print.

CLI input

The user enters a command on the CLI, followed possibly by arguments, typically separated by spaces, and presses Enter.

The CLI device parses the first word of this input line and compares that against the list of commands you have registered. If it finds a matching command, it sends the full input string using the dn_cli_notifMsg_t notification with dn_cli_notifMsg_t::type set to DN_CLI_NOTIF_INPUT, over the channel you specified when registering that command.

If the user types the string "help" followed by a command name, the CLI device will similarly find the corresponding registered command and issue a dn_cli_notifMsg_t notification with dn_cli_notifMsg_t::type set to DN_CLI_NOTIF_HELP. It is up to your application to print the correct output over CLI, typically its usage.

User Access Levels

The CLI device includes built-in support for various user levels, listed in dn_cli_access_t. The CLI device only allows access to commands registered with the current user level, or below. Similarly, when a user types help on the CLI interface, this component will list the commands available, at the current user level or below.

When opening the CLI device, the (default) access level is DN_CLI_ACCESS_LOGIN. The CLI commands registered by the protocol stack require access level DN_CLI_ACCESS_USER or higher.

When registering a command, your application needs to specify a minimum user access level (part of dn_cli_registerCmd_t).

Changing the current user level is done by calling the dn_ioctl() function with the following parameters:

Parameter Name Parameter Value
device DN_CLI_DEV_ID
request DN_IOCTL_CLI_CHANGE_ACCESS
args Points to a pre-populated dn_cli_chAccessCmd_t structure.
argLen Size of dn_cli_chAccessCmd_t.

It is up to your application to determine when to change the user access level. You could, for example, register a "login" command allowing a user to log in, possibly by specifying a password as argument to authenticate. Your application then could switch the user access level if the user entered the correct password. Similarly, a "logout" CLI command could reset the access level to the lowest one.

Changing Output Mode

The CLI device supports blocking and non-blocking output modes:

  • in blocking mode, a call to dn_write() blocks until the bytes are written in the CLI output buffer and are scheduled to be printed over the serial port.
  • in non-blocking mode, a call to dn_write() returns immediately, even when the serial transmit buffer is full. If the serial transmit buffer is full, the bytes are dropped and not printed over the serial port. The characters "<tt>...</tt>" are printed on the serial port to indicate lost output to the user.

To change this mode, you application needs to call the dn_ioctl() function with the following parameters:

Parameter Name Parameter Value
device DN_CLI_DEV_ID
request DN_IOCTL_CLI_SET_OUTPUT_MODE
args Points to a pre-populated dn_cli_setOutMode_t structure.
argLen Size of dn_cli_setOutMode_t.

Close the device

Close the device by calling the dn_close() function. This releases the device and allows a (possibly different) task to open and use it.
