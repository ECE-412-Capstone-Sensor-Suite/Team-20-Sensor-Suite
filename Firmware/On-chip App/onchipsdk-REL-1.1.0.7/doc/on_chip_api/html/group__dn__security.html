Description

Encrypting/decrypting arbitrary payload.

The LTC8500 chip contains an AES block cipher core, as well as the necessary support hardware for full CCM* generic combined encryption and authentication block cipher mode, as defined in the IEEE802.15.4 standard.

This software component allows your application to initialize and set up the hardware module, which executes all the calculations. It also allows you to use the AES CTR mode.

Note
The On-Chip Firmware Development Guide contains code snippets and sample applications, some of which use this device.

dn_error_t dn_sec_aesCbcMac ( INT8U *  inBuf,
INT16U  len,
INT8U *  key,
INT8U *  iv,
INT8U *  mic 
)

Run AES CBC-MAC on a payload, and generate a MIC.

Parameters
[in]inBufInput buffer to authenticate.
[in]lenLength of the input buffer.
[in]key128-bit key.
[in]ivInitialization vector.
[out]micPointer to a 16-byte MIC buffer that will be filled by this function.
Note
Both Message Authentication Code (MAC) and Message Integrity Code (MIC) refer to the same value calculated by this function. Both terms are used in the literature.

Strictly speaking, AES CBC-MAC does not use an initialization vector. If you're supplying the entire payload to the dn_sec_aesCbcMac() function, you should set the iv parameter to be all 0s. If, however, you desire to calculate MIC over a non-contiguous payload, you can call dn_sec_aesCbcMac() repeatedly. The first call should start with iv set to all 0s, and all subsequent calls should use the result MIC of previous call as iv input.

For example, if your application data is split in two, you can use this function to compute the MIC on the first half with iv set to 0s, store the resulting MIC, and use that as the iv when resuming the calculation on the second half.

Note
All pointers must point to internal RAM memory.
Returns
DN_ERR_NONE if the functions completed successfully.

Perform an in-place CCM encrypt operation on the specified payload.

Parameters
[in]optCCM options.
[in]aBufData to be authenticated, but not encrypted.
[in]aLenNumber of bytes in the buffer pointed to by aBuf.
[in,out]mBufData to both authenticate and encrypt.
[in]mLenNumber of bytes in the buffer pointed to by mBuf.
[in]key128-bit key.
[in]nonceNonce buffer, must be at least (15-opt->L) bytes.
[out]micBuffer to write the resulting MIC to. This buffer must contain at least opt->M bytes.
Note
All pointers must refer to internal RAM memory.
Postcondition
This function will modify the contents of the location pointed at by the mBuf parameter.
Returns
DN_ERR_NONE if the functions completed successfully.
DN_ERR_SIZE if mLen is too large, i.e. more than 128 bytes.

Perform an in-place CCM decrypt operation on the specified payload.

Parameters
[in]optCCM options.
[in]aBufData to authenticate, but not decrypt.
[in]aLenNumber of bytes in the buffer pointed to by aBuf.
[in,out]mBufData to both authenticate and decrypt.
[in]mLenNumber of bytes in the buffer pointed to by mBuf.
[in]key128-bit key.
[in]nonceNonce buffer, must be at least (15-opt->L) bytes.
[in]micMIC to validate. This buffer must contain at least opt->M bytes.
Note
All pointers must point to internal RAM memory.
Postcondition
This function will modify the contents of the location pointed at by the mBuf parameter.
Returns
DN_ERR_NONE if the functions completed successfully, i.e. the packet was decrypted and validated.
DN_ERR_SIZE if mLen is too large, i.e. more than 128 bytes.
DN_ERR_ERROR if MIC validation failed.
